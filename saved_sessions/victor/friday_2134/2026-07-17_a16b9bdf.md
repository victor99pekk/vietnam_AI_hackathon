## What the Pipeline Does

It takes raw, unstructured text and turns it into a structured, queryable knowledge graph. Here's how each stage works on the Alan Turing biography:

### Stage 1: Ingest & Clean
Reads `alan_turing.txt` and normalizes whitespace, quotes, dashes, and strips non-printable characters.

### Stage 2: Dedup & Quality Filter
Checks that the text is long enough, not gibberish, and not a near-duplicate of another document (Marie Curie's bio is different, so both pass).

### Stage 3: Extract (Entities + Relations)
spaCy scans the text and identifies:

| Entity | Label |
|---|---|
| Alan Turing | `PERSON` |
| London | `GPE` (place) |
| King's College | `ORG` |
| Princeton University | `ORG` |
| Bletchley Park | `GPE` |

Then the relation extractor finds co-occurring entities in the same sentence and infers predicates:

```
(Alan Turing, lives_in, London)
(Alan Turing, studied_at, King's College)
(Alan Turing, associated_with, Enigma)
```

### Stage 4: Resolve
Merges duplicate mentions — if "Turing" and "Alan Turing" both appear, they become one canonical node with both mentions tracked.

### Stage 5: Build, Evaluate & Export
Constructs a directed graph, scores it across 7 quality dimensions, and exports to JSON + GraphML.

---

## Features of the Final Knowledge Graph

From the sample run: **77 nodes, 544 edges, 566 triples** from just two short biographies.

| Feature | What you get |
|---|---|
| **Structured triples** | Every fact is a `(subject, predicate, object)` triple — directly usable for LLM fine-tuning |
| **Entity labels** | Nodes are typed: `PERSON`, `ORG`, `GPE`, `DATE`, `WORK_OF_ART`, `CONCEPT` |
| **Canonical entities** | Multiple mentions like "Turing" / "Alan Turing" resolve to one node |
| **Ontology-validated** | Optional schema enforcement — checks that `PERSON → works_at → ORG` pattern is followed |

While a **text chunk node** is a container for raw text, an **entity node** represents a clean, resolved "thing" in the real world (e.g., a specific person, product, concept, or disease).

Because your LLM or GraphRAG system will query these entities to piece together complex answers, your entity nodes need attributes that support both **logical graph queries** and **semantic language understanding**.

Here are the essential attributes that every robust entity node should have:

## 1. Identification & Nomenclature
These attributes ensure that the system knows exactly what the entity is and can recognize it even when it is referred to by different names in raw text.

- **`id` (String/URI):** A unique, immutable identifier. Ideally, this follows a standard URI format or a structured string (e.g., `person:albert_einstein` or `prod:router_ac1200`).
- **`name` / `displayName` (String):** The primary, canonical name of the entity (e.g., `"Albert Einstein"` or `"Ibuprofen"`). This is what the LLM will output.
- **`aliases` (List of Strings):** Synonyms, acronyms, or common misspellings (e.g., `["Einstein", "A. Einstein", "Father of Relativity"]`).

- *Why it's crucial for LLMs:* If a user searches for "A. Einstein," the system uses this attribute to map the query to the correct `Albert Einstein` node (Entity Resolution).

## 2. Semantic Grounding (For the LLM)
These properties give the LLM immediate, high-quality context about the entity without forcing the system to traverse the entire graph to figure out what the node means.

- **`description` / `definition` (String):** A brief, encyclopedic summary of what the entity is (e.g., `"A theoretical physicist who developed the theory of relativity, one of the two pillars of modern physics."`).

- *Why it's crucial for LLMs:* During RAG, you can inject this description directly into the prompt. It acts as an instant "cheat sheet" for the LLM.
- **`type` / `category` (String or Label):** The high-level class of the entity (e.g., `"Person"`, `"ChemicalCompound"`, `"SoftwareBug"`). While often represented as graph labels, having it as a property makes filtering fast and simple.

## 3. Vector & Graph Intelligence (For Search)
To make your graph searchable by mathematical algorithms, you need to store metadata that helps rank and retrieve the most important nodes.

- **`embedding` (Array of Floats):** The vector representation of the entity. Usually, this is generated by embedding the entity’s `name` and `description` together.

- *Why it's crucial:* This allows you to do "Entity-to-Entity" semantic searches.
- **`importanceScore` / `pageRank` (Float):** A calculated metric showing how "central" or "popular" this entity is in your graph.

- *Why it's crucial:* If a user queries a broad term, you can use this score to prioritize showing highly connected, important entities over obscure, weakly connected ones.

## 4. Provenance & Lifecycle (For Maintenance)
Graphs are living datasets. To keep them from deteriorating, you must track where your entities came from and when they were last updated.

- **`source` (String/List):** Where this entity was extracted from (e.g., `"Wikidata"`, `"Internal PDF v4"`, or a list of Chunk IDs).
- **`confidenceScore` (Float):** If the entity was extracted automatically by an LLM or an NLP pipeline, this tracks how confident the algorithm was that this entity actually exists (e.g., `0.92`).
- **`createdAt` / `updatedAt` (Timestamp):** Crucial for cache-busting and data synchronization.
